# SPI子系统框架
需明确，SPI子系统是通用设备驱动模型的一层特化
实质上，全都跑在同一个内核核心框架上
除开硬件和应用层，可分为三层
设备驱动层：应用SPI子系统框架
核心层：构建SPI子系统框架
控制器驱动层：适配SPI子系统框架
核心层 spi.c spi.h
spi.c：
初始化spi子系统，注册spi_bus，注册spi_master_class
提供设备驱动对总线操作的API
提供控制层注册master的API以及回调函数
spi.h
定义了一些数据结构，struct spi_master,struct spi_device,struct spi_message,struct spi_transfer
实现了一些简单函数
核心层的作用
对上层使用者，设备驱动层，提供标准spi收发API，以及设备注册函数
对下层适配者，控制器驱动层，提供控制器注册API，以及需要控制器驱动实现的回调函数
# 核心层
## 初始化
### spi_init
spi子系统初始化函数，注册了总线和master class
成功后在/sys/bus/下可以看到spi文件目录，在/sys/class/下可以看到spi_master文件目录
### spi从设备和驱动匹配
spi总线和类注册后，当有驱动和设备匹配上就会调用spi_match_device，也就是spi_bus_type.match函数
1.通过of_match_table的compatible和设备树匹配
2.通过acpi_match_table的compatible和device的of_node的compatible匹配
3.通过驱动和设备的id_table去匹配
4.最后通过驱动和设备的名字去匹配
## 核心层的API
### spi_register_master（供HOST层调用）
1.of_spi_register_master，根据设备树节点中的“cs-gpio“，向struct spi_master添加gpio cs引脚
2.device_add将设备注册到总线驱动设备模型中
3.如果控制器驱动没有实现transfer函数，则初始化发送队列 spi_master_initialize_queue，核心层默认填充transfer函数
`c
4.spi_match_master_to_boardinfo(老的设备发现方法)遍历所有的spi_board_info数据结构，并注册spi_device
`
5.of_register_spi_devices（新的设备树方法），遍历spi控制器节点下所有子节点，并注册对应的spi_device设备
#### spi_master_initialize_queue
可以看到对master成员函数赋值，transfer或transfer_one_message赋值成默认函数，然后调用spi_init_queue和spi_start_queue初始化工作线程。spi_init_queue最大作用是建立一个内核工作线程。
spi_init_queue函数先初始化kthread_worker，为kthread_worker创建一个内核线程，随后初始化kthread_work，设置work执行函数，work执行函数为spi_pump_messages
spi_start_queue函数用于唤醒工作线程
队列化的相关工作已经完成，系统等待message请求被发起，然后工作线程中处理message的传送工作
### spi_message_init_（供从设备调用）
对message进行初始化
### spi_message_add_tail(供从设备使用)
就是把spi_transfer这个buffer添加到spi_message传输链表中
### spi_async(供从设备调用)
`c
用于发起数据传输，从spi_async->__spi_async->master->transfer。asyn是异步执行，不会等待传输是否完成，直接返回。
`
如果控制器驱动没有实现transfer，内核会自己填充默认的transfer函数spi_queued_transfer
#### spi_queued_transfer
默认spi_queued_transfer就是把message添加到master的发送链表中。接下来就交给工作服务线程__spi_pump_message去处理message。
### spi_sync(供从设备调用)
spi_sync用来同步传输message，完成传输后调spi_complete唤醒等待的线程
如果控制器驱动没有实现transfer，内核会自己填充默认的transfer函数spi_queued_transfer
把message添加到master的发送链表中，调用工作服务线程__spi_pump_message去处理message
最后调用wait_for_completion等待传输完成
### spi_bitbang_start（供HOST层调用）
定义在spi.bitbang.c中
`c
如果 master 驱动没有显式设置 master->mode_bits，则设置为默认的模式位，包括CPOL,CPHA,flag（位移传输结构体中标记位）
`
检查主设备结构体中的传输函数是否已经定义，如果已定义则返回错误码-EINVAL。
设置主设备结构体中的准备硬件传输函数、释放硬件传输函数、单次传输函数和片选信号控制函数，分别对应位移传输控制结构体中的对应函数。
如果位移传输控制结构体中的数据缓冲区传输函数未定义，则设置使用 DMA 标志为 0，并将数据缓冲区传输函数设置为默认的位移传输函数 spi_bitbang_bufs。如果主设备结构体中的设置函数未定义，则设置使用默认的设置函数 spi_bitbang_setup 和清理函数 spi_bitbang_cleanup。
注册 SPI 主设备，将其添加到系统中。
# 控制器驱动层
控制器驱动主要用来实现spi_master中的成员函数，如transfer，transfer_one_message。
基本流程是：申请spi_master，初始化spi_master，向Linux内核注册spi_master
## SPI控制器驱动API
### spi_alloc_master
申请spi控制器内存
dev：设备，一般是platform_device中的dev成员变量
size：私有数据大小，可以通过spi_master_get_fevdata函数获取这些数据
返回值：申请到的spi_master
### spi_master_put
spi_master的释放通过spi_master_put函数来完成，当我们删除一个 SPI 主机驱动的时候就 需要释放掉前面申请的spi_master
### spi_register_master
当spi_master初始化完成后需要将其注册到Linux内核，spi_register_master注册spi控制器
### spi_unregister_master
spi控制器卸载

### spi_imx_setupxfer(设置位宽和配置控制器)
在 bitbang 框架的 spi_bitbang_transfer_one_message 中，每个 transfer 执行前都会调一次 setup_transfer（即 spi_imx_setupxfer）。它的职责是针对当前 transfer 的速率和字长，配置好硬件，准备好收发函数。
`c
1. 确定这次传输的速率和字长
 2. 根据字长选择 tx/rx 函数指针（u8/u16/u32）
 3. 判断走 PIO 还是 DMA
 4. 如果走 DMA，配置 DMA 参数
 5. 调 devtype_data->config 写硬件寄存器
`
设置 spi_imx 的 tx 和 rx 传输函数。根据要发送的数据数据位宽的不 同，分别有 8 位、16 位和 32 位的发送函数
readl将要接收的数据值读到 ECSPI 的 MXC_CSPIRXDATA 寄存器里面去
writel将要发送的数据值写入到 ECSPI 的 TXDATA 寄存器里面去

static int mx51_ecspi_config(struct spi_device *spi,
struct spi_imx_config *config){
`c
struct spi_imx_data *spi_imx = spi_master_get_devdata(spi->master);
`
u32 ctrl = MX51_ECSPI_CTRL_ENABLE;
u32 clk = config->speed_hz, delay, reg;
u32 cfg = readl(spi_imx->base + MX51_ECSPI_CONFIG);
int tx_wml = 0;
/*
`c
* The hardware seems to have a race condition when changing modes. The
* current assumption is that the selection of the channel arrives
* earlier in the hardware than the mode bits when they are written at
`
* the same time.
`c
* So set master mode for all channels as we do not support slave mode.
`
*/
ctrl |= MX51_ECSPI_CTRL_MODE_MASK;
/* set clock speed */
ctrl |= mx51_ecspi_clkdiv(spi_imx, config->speed_hz, &clk);
spi_imx->spi_bus_clk = clk;
/* set chip select to use */
ctrl |= MX51_ECSPI_CTRL_CS(spi->chip_select);
ctrl |= (config->bpw - 1) << MX51_ECSPI_CTRL_BL_OFFSET;
cfg |= MX51_ECSPI_CONFIG_SBBCTRL(spi->chip_select);
if (spi->mode & SPI_CPHA)
cfg |= MX51_ECSPI_CONFIG_SCLKPHA(spi->chip_select);
else
cfg &= ~MX51_ECSPI_CONFIG_SCLKPHA(spi->chip_select);
if (spi->mode & SPI_CPOL) {
cfg |= MX51_ECSPI_CONFIG_SCLKPOL(spi->chip_select);
cfg |= MX51_ECSPI_CONFIG_SCLKCTL(spi->chip_select);
} else {
cfg &= ~MX51_ECSPI_CONFIG_SCLKPOL(spi->chip_select);
cfg &= ~MX51_ECSPI_CONFIG_SCLKCTL(spi->chip_select);
}
if (spi->mode & SPI_CS_HIGH)
cfg |= MX51_ECSPI_CONFIG_SSBPOL(spi->chip_select);
else
cfg &= ~MX51_ECSPI_CONFIG_SSBPOL(spi->chip_select);
if (spi_imx->usedma)
ctrl |= MX51_ECSPI_CTRL_SMC;
`c
/* CTRL register always go first to bring out controller from reset */
`
writel(ctrl, spi_imx->base + MX51_ECSPI_CTRL);
reg = readl(spi_imx->base + MX51_ECSPI_TESTREG);
if (spi->mode & SPI_LOOP)
reg |= MX51_ECSPI_TESTREG_LBC;
else
reg &= ~MX51_ECSPI_TESTREG_LBC;
writel(reg, spi_imx->base + MX51_ECSPI_TESTREG);
writel(cfg, spi_imx->base + MX51_ECSPI_CONFIG);
/*
`c
* Wait until the changes in the configuration register CONFIGREG
* propagate into the hardware. It takes exactly one tick of the
* SCLK clock, but we will wait two SCLK clock just to be sure. The
* effect of the delay it takes for the hardware to apply changes
* is noticable if the SCLK clock run very slow. In such a case, if
* the polarity of SCLK should be inverted, the GPIO ChipSelect might
* be asserted before the SCLK polarity changes, which would disrupt
* the SPI communication as the device on the other end would consider
`
* the change of SCLK polarity as a clock tick already.
*/
delay = (2 * 1000000) / clk;
if (likely(delay < 10))	/* SCLK is faster than 100 kHz */
udelay(delay);
else			/* SCLK is _very_ slow */
usleep_range(delay, delay + 10);
/*
* Configure the DMA register: setup the watermark
* and enable DMA request.
*/
if (spi_imx->devtype_data->devtype == IMX6UL_ECSPI)
tx_wml = spi_imx->wml / 2;
writel(MX51_ECSPI_DMA_RX_WML(spi_imx->wml) |
MX51_ECSPI_DMA_TX_WML(tx_wml) |
MX51_ECSPI_DMA_RXT_WML(spi_imx->wml) |
MX51_ECSPI_DMA_TEDEN | MX51_ECSPI_DMA_RXDEN |
MX51_ECSPI_DMA_RXTDEN, spi_imx->base + MX51_ECSPI_DMA);
return 0;
}

### spi_imx_transfer(数据收发)
决定PIO还是DMA
补充
PIO（Programmed I/O，程序控制输入输出） 和 DMA（Direct Memory Access，直接内存访问） 是两种 CPU 搬运数据的方式。
PIO 传输
CPU 亲自参与每个字节的搬运。
以 spi-imx 的 PIO 路径为例：
// 写一个字节到 TXFIFO——CPU 亲手写寄存器
spi_imx_buf_tx_u8:
val = *(u8*)spi_imx->tx_buf;        // CPU 从内存读
writel(val, base + MXC_CSPITXDATA);  // CPU 写寄存器
// 读一个字节从 RXFIFO——CPU 亲手读寄存器
spi_imx_buf_rx_u8:
val = readl(base + MXC_CSPIRXDATA);  // CPU 读寄存器
*(u8*)spi_imx->rx_buf = val;         // CPU 写内存
PIO 传输中，每发一个字节：CPU 读内存 → CPU 写 TXFIFO → 等待中断 → CPU 读 RXFIFO → CPU 写内存。全程 CPU 都在忙。
PIO 的典型时序图：
CPU   ──┬──────┬──────┬──────┬──────┬──
写     等     读     写
FIFO  中断    FIFO   FIFO
DMA 传输
CPU 告诉 DMA 引擎"帮我把这段内存的数据搬过去"，然后去做别的事。DMA 引擎自己读内存、写 FIFO、读 FIFO、写内存。
以 spi-imx 的 DMA 路径为例：
`c
// CPU 做的事只有这些：dmaengine_prep_slave_sg(master->dma_tx, tx_sg, ...);  // 准备 DMA 描述符dmaengine_submit(desc_tx);                              // 提交dma_async_issue_pending(master->dma_tx);                // 启动
`
// 然后 CPU 去做别的事了// DMA 引擎自动搬运数据
DMA 传输中，CPU 只做设置和收尾，中间的每一个字节搬运都由 DMA 引擎独立完成。
DMA 的典型时序图：
CPU   ──┬─────────────────────────┬──
设置DMA                   等待完成
│                         │
DMA     ──┬──┬──┬──┬──┬──┬──┬──┬──
读 写 读 写 读 写 读 写
内存 FIFO ... 内存 FIFO
关键区别
SPI 中怎么决定走 PIO 还是 DMA
// spi_imx_setupxfer 中
if (spi_imx_can_dma(spi_imx->bitbang.master, spi, t))
spi_imx->usedma = 1;
else
spi_imx->usedma = 0;
// spi_imx_transfer 中
if (spi_imx->usedma)
return spi_imx_dma_transfer(spi_imx, transfer);
else
return spi_imx_pio_transfer(spi, transfer);
can_dma 的判断通常是传输长度是否超过某个阈值——几个字节用 PIO，几十 KB 用 DMA。

# SPI设备驱动层
设备驱动核心就分这 5 大块，任何一个驱动都是这个骨架：
块一：驱动对象声明
告诉内核"我是谁、我在哪条总线上走、怎么匹配到我"。
static struct spi_driver my_driver = {
.driver = {
.name           = "mydev",         // 名字，匹配用
.of_match_table = my_of_match,     // DT 匹配表
.owner          = THIS_MODULE,     // 模块引用
},
.probe  = my_probe,
.remove = my_remove,
};
`c
// 入口/出口二选一：module_spi_driver(my_driver);        // 宏版// 或：module_init(my_init); module_exit(my_exit);  // 手写版
`
块二：probe — 初始化
匹配成功时调用，做资源申请和硬件初始化。标准步骤：
① devm_kzalloc 分配私有数据结构
② spi_setup() 配置控制器（模式/速率/字长）
③ devm_request_irq() 注册中断（如果有）
④ 注册用户态接口（device_create_file / cdev_add / input_register...）
⑤ spi_set_drvdata(spi, data) 保存私有数据
块三：数据传输 — 读写操作
业务逻辑的核心，用 SPI 的传输 API 和硬件交互：
`c
spi_write() / spi_read()           半双工spi_write_then_read()              先写后读（寄存器读写最常用）spi_sync() / spi_message()         全双工、多 transferspi_async()                        异步传输
`
块四：用户态接口
驱动和用户态程序之间的桥：
sysfs：device_create_file，最轻量，cat/echo 就能交互
字符设备：alloc_chrdev_region + cdev_init + cdev_add，完整 read/write/ioctl
内核子系统：input / IIO / hwmon 等，享受子系统的标准 API 和工具链
块五：remove — 清理
probe 的逆操作，资源交还：
① devm_ 的资源自动释放
② device_remove_file / device_destroy
③ 非 devm_ 的资源手动释放（kfree、free_irq）
其实就是"注册 → probe → 业务 → 接口 → remove"五段论，无论你写 SPI、I2C、PCI 还是 Platform 驱动，骨架都是这个结构，换的只是总线和传输 API。

## SPI从设备示例
### spidev SPI万能从设备驱动
`c
spidev 实际上是**最上层的 SPI 设备驱动**，但它不操作具体硬件——它把用户请求翻译成 `spi_message` 交给核心层。它注册的是 `spi_driver`，也是 `chrdev`（字符设备）。
`
万能SPI从设备驱动对应spidev，不用为每个spi从设备去写一个驱动，linux内核有一个万能通用的从设备驱动。
为什么说spidev.c是一个通用的从设备驱动。
spidev不是专门针对某一SPI硬件设备做的驱动，只是简单的注册字符设备，用户空间通过read、write、ioctl，直接对spi进行操作，相当于是用户空间和spi设备的桥梁。用户空间操作时，打开设备节点，然后通过IOCTL设置模式、速度等参数，然后就可以调用read write进行操作了。
`c
一个普通的 SPI 设备驱动（比如温度传感器驱动）知道设备的具体协议——先发什么命令、等待多久、读几个字节、怎么解析。而 spidev **完全不知道协议**——它只是把用户的 raw 数据原样传给核心层，再把核心层返回的 raw 数据还给用户。
所以 spidev 是一个**通用驱动**（generic driver），不绑定任何具体硬件。这也意味着：**使用 spidev 的应用层代码必须自己实现 SPI 设备的协议**
`

### 补充
驱动有两种分类方式，不是互斥的，而是交叉的：
维度一：按"设备类型"分 — 3 大类
这是最传统的分类，来自 Unix 的"一切皆文件"哲学。
1. 字符设备（Character Device）
数据流是字节流，顺序访问，不支持随机定位
通过 /dev 下的设备节点访问
注册方式：register_chrdev 或 alloc_chrdev_region + cdev_add
file_operations：open/read/write/ioctl/release
spidev 就属于这一类
典型例子：串口（tty）、SPI/I2C 用户态接口、GPIO、LED
2. 块设备（Block Device）
以数据块为单位访问，支持随机定位（lseek）
有缓存层（page cache），通过文件系统挂载
注册方式：register_blkdev
block_device_operations：open/release/rw_page/ioctl
典型例子：硬盘（sda）、SSD（nvme0）、SD 卡（mmcblk0）
3. 网络设备（Network Device）
不以文件节点访问，通过 socket 接口
数据包为单位（不是字节流也不是块）
注册方式：register_netdev
操作集：net_device_ops（ndo_open/ndo_start_xmit/ndo_do_ioctl）
典型例子：eth0、wlan0、lo
维度二：按"连接总线"分 — 驱动模型分类
这是 Linux 设备驱动模型（driver model）的分类，取决于设备挂在哪条总线上。
1. Platform Driver
挂载在虚拟的 platform_bus 上
对应 SoC 内部集成的外设（没有物理枚举总线）
匹配方式：设备树 compatible 或 platform ID table
spi-imx.c 就是 platform_driver（SPI 控制器是 SoC 内部外设）
注册：platform_driver_register
2. SPI Driver
挂载在 spi_bus_type 上
对应 SPI 总线上的外设芯片
匹配方式：DT compatible / id_table / name
spidev.c 注册的是 spi_driver
注册：spi_register_driver
3. I2C Driver
挂载在 i2c_bus_type 上
注册：i2c_add_driver
4. PCI Driver
挂载在 pci_bus_type 上
匹配方式：Vendor ID + Device ID
5. USB Driver
挂载在 usb_bus_type 上
另外还有 MMC、SERIO、MDIO 等等，都是按总线划分。
两个维度交叉看：spidev 到底属于哪种？
char    block    net
platform  │       │       │
spi      spidev   │       │
i2c       │       │       │
pci       │      nvme     │
usb      ttyUSB   │      wlan
spidev 是交叉点：它既是 spi_driver（按总线分），又是字符设备（按设备类型分）。
这个理解很关键：
spi_driver 的身份让它能被 SPI 总线识别、匹配、probe——解决"怎么找到硬件"的问题
chrdev 的身份让它能暴露 /dev/spidev0.0 给用户态——解决"怎么和用户交互"的问题
一个驱动可以同时属于多个类型。比如：
i2c-dev：i2c_driver + chrdev（类似 spidev 的角色）
nvme：pci_driver + block driver
gpiolib：platform_driver + chrdev（/dev/gpiochip0）
# 重要数据结构
## 一、通用设备驱动模型（drivers/base/）
Linux 设备模型的三大支柱，所有总线/设备/驱动都基于它们。
### struct bus_type — 总线抽象
struct bus_type {
`c
const char *name;                              // 总线名，如 "spi"
int (*match)(struct device *dev, struct device_driver *drv);  // 匹配函数
`
int (*probe)(struct device *dev);
int (*remove)(struct device *dev);
`c
struct attribute_group **bus_groups;           // 总线级 sysfs 属性
`
struct subsys_private *p;                      // 内部：设备/驱动链表
};
关键：match 是总线的核心职责。spi 总线的 spi_match_device 按 DT → ACPI → id_table → name 顺序匹配。总线内部维护了两个链表——这条总线上的所有设备和所有驱动。
### struct device — 设备抽象
struct device {
const char *init_name;          // 设备名
struct bus_type *bus;           // 挂在哪条总线上
struct device *parent;          // 父设备
struct device_node *of_node;    // DT 节点
void *driver_data;              // 驱动私有数据 ← 常用！
struct kobject kobj;            // sysfs 入口
};
关键：device 是"对象"——总线上挂的是 device，驱动 match 的是 device，probe 传入的也是 device。driver_data 是最常用的字段——通过 dev_set_drvdata/dev_get_drvdata 访问。在 SPI 里包装为 spi_set_drvdata/spi_get_drvdata。
### struct device_driver — 驱动抽象
struct device_driver {
const char *name;
struct bus_type *bus;
struct module *owner;
const struct of_device_id *of_match_table;
int (*probe)(struct device *dev);
int (*remove)(struct device *dev);
void (*shutdown)(struct device *dev);
int (*suspend)(struct device *dev, pm_message_t state);
int (*resume)(struct device *dev);
const struct dev_pm_ops *pm;
struct driver_private *p;       // 内部：sysfs/klist
};
### 三者的关系
bus_type 维护两个链表：
├── 设备链表：struct device 通过 bus->p->devices_kset 链接
└── 驱动链表：struct device_driver 通过 bus->p->drivers_kset 链接
匹配时：
bus->match(device, driver) → 成功 → driver->probe(device)
## 二、SPI 核心层（spi.h / spi.c）
### struct spi_master — SPI 控制器抽象（核心层最重要的结构）
struct spi_master {
`c
struct device dev;                  // 内嵌 device，挂到 spi_master class
`
/* 参数 */
s16 bus_num;                        // 总线编号
u16 num_chipselect;                 // CS 数量
u32 mode_bits;                      // 支持的模式位
u32 bits_per_word_mask;             // 支持的 bpw 掩码
u32 min_speed_hz, max_speed_hz;     // 速率范围
/* 消息泵 */
struct list_head queue;             // 消息队列
struct kthread_worker *kworker;     // kthread 工作者
struct task_struct *kworker_task;   // kthread 任务
/* DMA */
struct dma_chan *dma_tx, *dma_rx;
int (*can_dma)(...);
/* 回调 — host 层实现 */
int (*setup)(struct spi_device *spi);
int (*transfer_one_message)(...);   // 处理一个 message
int (*transfer_one)(...);           // 处理一个 transfer
int (*prepare_message)(...);        // message 前开时钟
int (*unprepare_message)(...);      // message 后关时钟
void (*cleanup)(...);
/* 设备自动注册 */
void *(*cs_gpios);                  // CS GPIO 列表
};
关键：spi_master 是 SPI 核心层与 HOST 控制器驱动之间的契约。核心层通过回调（transfer_one_message、setup）调用 HOST 层；HOST 层通过 spi_alloc_master 分配它，并填充回调。
### struct spi_device — SPI 设备抽象
struct spi_device {
struct device dev;                  // 内嵌 device，挂到 spi bus
struct spi_master *master;          // 所属控制器
u32 max_speed_hz;                   // 最大速率
u8 chip_select;                     // CS 号
u8 bits_per_word;                   // 字长
u16 mode;                           // SPI 模式
int irq;                            // 中断号（来自 DT）
void *controller_data;              // 控制器私有数据
const char *modalias;               // 用于匹配的名字
};
`c
关键：每个 SPI 外设芯片对应一个 spi_device。设备驱动通过 to_spi_device(dev) 从通用 struct device 获取它。
`
### struct spi_transfer — 单次传输段
struct spi_transfer {
const void *tx_buf;                 // 发送数据
void *rx_buf;                       // 接收数据
unsigned len;                       // 长度
u32 speed_hz;                       // 本次传输速率（可覆盖默认）
u16 delay_usecs;                    // 传输后延时
u8 bits_per_word;                   // 本次字长（可覆盖默认）
u8 cs_change;                       // 传输后是否切换 CS
u32 tx_nbits, rx_nbits;             // 多线模式（DUAL/QUAD）
};
### struct spi_message — SPI 事务
struct spi_message {
struct list_head transfers;         // transfer 链表
struct spi_device *spi;             // 目标设备
unsigned actual_length;             // 实际传输字节数
int status;                         // 传输状态
void (*complete)(void *context);    // 异步完成的回调
void *context;                      // 回调参数
};
关键：spi_message 包含多个 spi_transfer，在同一个 CS 周期内按序执行。这是 SPI 子系统最核心的设计——上层的设备驱动构造 message，下层的 HOST 驱动执行它。
### 四者关系
spi_master（控制器）
│
├── 管理多个 spi_device（挂载的外设）
│
└── 接收 spi_message（从设备驱动来）
└── 包含多个 spi_transfer
├── 每个 transfer 是一段连续数据流
└── 可单独配置速率/字长
## 三、HOST 控制器驱动层（spi-imx.c 为代表）
### struct spi_imx_data — 控制器私有数据
struct spi_imx_data {
struct spi_bitbang bitbang;             // bitbang 框架嵌入
/* 硬件资源 */
void __iomem *base;                     // 寄存器基址（ioremap 结果）
struct clk *clk_per;                    // 外设时钟
struct clk *clk_ipg;                    // 接口时钟
unsigned long spi_clk;                  // 输入时钟频率
/* 芯片类型 */
struct spi_imx_devtype_data *devtype_data;  // 芯片特定回调节点
/* 函数指针 */
`c
void (*tx)(struct spi_imx_data *);      // 写 TXFIFO 函数（u8/u16/u32）
`
void (*rx)(struct spi_imx_data *);      // 读 RXFIFO 函数
/* PIO 传输状态 */
const void *tx_buf;                     // 当前发送位置
void *rx_buf;                           // 当前接收位置
unsigned int count;                     // 剩余字节数
unsigned int txfifo;                    // 软件 FIFO 计数
/* 同步 */
struct completion xfer_done;            // 传输完成通知
/* DMA */
unsigned int wml;                       // FIFO 水线
struct completion dma_tx_completion;    // DMA TX 完成
struct completion dma_rx_completion;    // DMA RX 完成
unsigned int usedma;                    // 当前传输是否走 DMA
};
### struct spi_imx_devtype_data — 芯片差异抽象
struct spi_imx_devtype_data {
void (*intctrl)(struct spi_imx_data *, int);        // 中断控制
`c
int  (*config)(struct spi_device *, struct spi_imx_config *); // 配寄存器
void (*trigger)(struct spi_imx_data *);             // 触发 SCLK
int  (*rx_available)(struct spi_imx_data *);        // 检查 RX 数据
`
void (*reset)(struct spi_imx_data *);                // 复位
enum spi_imx_devtype devtype;                       // 芯片枚举
};
关键：通过函数指针表，同一个 spi-imx.c 支持 IMX1 ~ IMX6UL 五款芯片。probe 时根据 DT compatible 选择对应实例。
### struct spi_bitbang — bitbang 框架
struct spi_bitbang {
`c
struct spi_master *master;                      // 关联的 master
`
int (*chipselect)(struct spi_device *, int);     // CS 控制回调
int (*txrx_bufs)(struct spi_device *,            // 收发回调
struct spi_transfer *);
int (*setup_transfer)(struct spi_device *,       // 传输配置回调
struct spi_transfer *);
};
关键：spi_imx 通过 bitbang 框架接入核心层。bitbang 框架替它实现了 transfer_one_message（循环遍历 transfer + 拉 CS + 调 txrx_bufs + 释放 CS）。
### struct spi_imx_config — 单次传输配置
struct spi_imx_config {
unsigned int speed_hz;  // 传输速率
unsigned int bpw;       // 位宽
};
